Goal:

To create a dockerized environment for Loot Survivor that would:

1. Package everything needed to run the game:
   - Katana (local Starknet node)
   - Indexer for game state
   - The game contracts
   - Our agent system

2. Make it easy for people to:
   - Spin up their own local environment
   - Experiment with different agent strategies
   - Design and test different context files
   - Run multiple experiments quickly

This would essentially create a "Loot Survivor AI Arena" where people could:
- Develop and test different AI strategies
- Compare different approaches
- Share and iterate on context files
- Benchmark agent performance

Project Context & Key Concepts:
1. Game Overview
   - Loot Survivor is an on-chain game running on Starknet
   - Players control adventurers who explore, fight beasts, and collect loot
   - Game state is managed through smart contracts written in Cairo
   - Actions include: exploring, fighting, upgrading stats, buying/using items

2. System Architecture
   - Katana: Local Starknet node for development (like Ganache for Ethereum)
   - MongoDB: Stores indexed game state and events
   - Redis: Caches frequently accessed data
   - GraphQL API: Provides structured queries for game state
   - Indexers: Transform on-chain events into queryable data
   - Agent System: AI-powered decision making using Chain of Thought

3. Key Components
   - Chain of Thought: Our core decision-making system
     - Uses LLM to process game state and make decisions
     - Guided by context files that define strategy and rules
     - Executes actions through contract interactions
   
   - Context Files: Strategy definitions for agents
     - Define game rules and mechanics
     - Specify decision-making priorities
     - Include parsing logic for game state
     - Define action validation rules

   - Indexers: Event processors
     - Listen for contract events
     - Update MongoDB with game state
     - Enable efficient querying
     - Support real-time updates

4. Current State
   - Basic infrastructure is containerized and running
   - Services can communicate with each other
   - Environment variables and networking configured
   - Missing: Contract deployment, indexer setup, agent integration

5. Development Flow
   - Contracts deployed to local Katana node
   - Events captured by indexers and stored in MongoDB
   - GraphQL API provides query interface
   - Agents read state and execute actions
   - All running in isolated Docker environment

Critical Challenges:

1. State Synchronization (Highest Priority)
   - Challenge: Maintaining consistent game state across all components
     * Contract state must sync with MongoDB
     * Indexers must process events in correct order
     * Agent decisions must be based on current state
     * Multiple agents may act simultaneously
   
   - Impact:
     * Incorrect state = wrong agent decisions
     * Missing events = incomplete game history
     * Race conditions = invalid actions
     * State lag = poor agent performance

   - Mitigation Strategy:
     * Implement event versioning and ordering
     * Add state verification checkpoints
     * Create rollback mechanisms
     * Build robust error handling
     * Add state validation between components

2. Secondary Challenges:
   
   a. Contract Deployment Reliability
      - Ensuring consistent contract deployment across fresh environments
      - Managing contract addresses and configuration
      - Handling failed deployments and retries
   
   b. Agent Performance Optimization
      - Reducing LLM latency for faster decisions
      - Optimizing context file processing
      - Managing memory usage for long games
   
   c. Development Experience
      - Making the system accessible to new developers
      - Providing clear debugging tools
      - Creating meaningful error messages
      - Simplifying the setup process

These challenges are interconnected - solving state synchronization will make other aspects like agent performance and development experience much more manageable.
